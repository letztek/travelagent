# **專案章程 (Project Charter)**

## **a) 專案名稱與目的 (Project Name and Purpose)**

* **專案名稱**：AI 智慧旅遊業務助理系統 (TravelAgent)  
* **專案目的**：  
  * 解決傳統旅遊業務在「需求彙整」與「初版行程規劃」上的重複性勞動。  
  * 透過生成式 AI 技術，將非結構化的客戶需求快速轉化為結構化的行程方案與簡報大綱，釋放資深顧問的時間以專注於客戶關係與複雜問題處理。

## **b) 專案目標與成功準則 (Project Objectives and Success Criteria)**

* **專案目標**：  
  * 開發一套 MVP (最小可行性產品)，實現「需求解析」至「行程規劃」的自動化。  
  * 提供「中介 Prompt 生成」功能，協助業務人員利用外部 AI 工具快速產出高品質簡報。  
* **成功準則 (Success Criteria)**：  
  1. **作業效率**：從「收到客戶需求」到「產出可供外部工具使用的 Briefing Prompt」，時間控制在 15 分鐘以內。  
  2. **資料結構化**：成功定義並實作「私房景點」與「客戶需求」的資料庫欄位 (Schema)，且系統能正確填入資訊。  
  3. **內容可用性**：AI 產出的行程邏輯（地理位置順序、停留時間）經資深顧問檢核，修改率低於 20%。  
  4. **簡報銜接度**：產出的 Prompt 複製到簡報生成工具（如 Gamma/Canva）後，能直接生成結構完整的行前說明會投影片。

## **c) 高層次需求 (High-Level Requirements) \- MVP 範疇**

本專案 MVP 聚焦於四大核心功能模組：

### **1\. 客戶需求自動彙整 (Requirement Parsing)**

* **功能描述**：系統能解析來自通訊軟體或 Email 的自然語言文字。  
* **產出**：填入「客戶需求輔助欄位」（包含：日期區間、人數結構、預算範圍、特殊飲食偏好、必去/避開景點、住宿星級要求）。

### **2\. 智慧行程規劃引擎 (Itinerary Planning Engine)**

* **資料庫準備**：完成「供應商/景點資料庫」的 Schema 設計（定義欄位如：景點標籤、建議停留時間、成本價、牌價等），暫不強制要求完整資料填充。  
* **邏輯處理**：  
  * 依據 GIS 地理位置邏輯，自動排序每日行程。  
  * 自動插入行政作業提醒（如：跨國移動時的簽證檢查點）。  
* **產出**：結構化的每日行程表（Day-by-Day Itinerary）。

### **3\. 行程變動與調整 (Dynamic Adjustment)**

* **功能描述**：提供對話式介面或簡易 UI，讓業務人員能快速調整行程（例如：「第三天下午改去博物館」）。  
* **連動性**：調整行程後，系統需自動重新計算相關的時間序與交通建議。

### **4\. 交付與 Briefing 輔助 (Delivery & Briefing Support)**

* **核心功能**：系統不直接產出最終 PPT，而是生成\*\*「結構化 Prompt」\*\*。  
* **Prompt 內容**：包含封面標題、每日行程亮點、注意事項、緊急聯絡資訊預留欄位。  
* **應用場景**：業務人員複製此 Prompt 至 AI 簡報工具，一鍵生成行前說明會簡報。

*(備註：自動化報價系統因涉及複雜商業邏輯與利潤計算，列入下一階段或僅在 MVP 中提供基礎成本加總功能，細節待定。)*

## **d) 假設與限制事項 (Assumptions and Constraints)**

* **假設 (Assumptions)**：  
  * 資深業務顧問能提供足夠的歷史行程案例，供 AI 學習或作為 Few-Shot Prompting 的範本。  
  * 外部 AI 簡報工具（如 Gamma）的 Prompt 格式維持穩定，不會在開發期間大幅變動。  
* **限制 (Constraints)**：  
  * **資料庫內容**：MVP 階段不包含完整的供應商資料庫，系統需具備容錯能力，允許在資料缺失時透過 LLM 的通用知識補位。  
  * **介面整合**：系統與外部簡報工具之間無 API 串接，需透過「人工複製貼上」進行資料傳遞。  
  * **金流處理**：MVP 不處理實際金流與發票開立，僅做數據紀錄。

## **e) 專案描述 (Project Description)**

本系統定位為旅遊業務的「數位助理」。系統負責處理資訊的結構化與草稿生成，將「從 0 到 1」的耗時工作自動化。業務人員則轉型為「編輯者」與「審核者」，利用系統產出的高品質 Prompt 驅動其他生產力工具，完成最終交付物的製作。

## **f) 高層次風險 (High-Level Risks)**

1. **Prompt 有效性風險**：生成的 Prompt 在不同簡報工具（Gemini vs Canva vs Seede）上效果不一。  
   * *緩解*：針對特定一款主流工具進行優化與測試。  
2. **行程邏輯斷裂**：AI 可能安排地理位置不順路的行程。  
   * *緩解*：在介面上保留人工拖拉或修改的彈性，不追求 100% 全自動。  
3. **需求理解誤差**：對於語意模糊的客戶需求（如「想要有放鬆的感覺」），AI 定義可能與業務認知不同。  
   * *緩解*：設計「確認反饋機制」，AI 在規劃前先列出它理解的重點供業務確認。

## **g) 里程碑摘要 (Summary Milestone Schedule)**

* **Sprint 1 (資料與架構)**：完成 Project Charter，設計客戶需求與景點資料庫的 Schema (欄位定義)。  
* **Sprint 2 (核心邏輯)**：開發需求解析模組與基礎行程規劃功能 (Text-to-Itinerary)。  
* **Sprint 3 (輸出整合)**：開發「行程調整 UI」與「簡報 Prompt 生成器」。  
* **Sprint 4 (驗收與優化)**：由資深顧問進行實際案例測試，調整 Prompt 參數以符合商業標準。

## **h) 預算摘要 (Summary Budget)**

* **開發成本**：內部人力 (Side Project)。  
* **營運成本**：  
  * LLM API (Gemini/OpenAI)：依用量計費，預估 MVP 測試期 \< $30 USD。  
  * 外部工具訂閱：如需使用進階版 AI 簡報工具的訂閱費用。

## **i) 專案核准需求 (Project Approval Requirements)**

* 系統需能成功解析至少 5 種不同類型的旅遊需求（如：家族旅遊、蜜月、員工旅遊等）。  
* 生成的 Briefing Prompt 經測試，能在外部工具中產出結構完整（含目錄、每日行程、注意事項）的投影片草稿。  
* **最終核決權人**：專案贊助人 (Sponsor)。
